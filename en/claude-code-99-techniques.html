Essential Commands
01 – 08
01
Plan Mode Shift + Tab
▼
Before any implementation, activate plan mode. Claude Code analyzes your codebase and creates an architectural plan without writing code. Review the plan, approve it, then go back to implementation. This single habit prevents more bugs than any other technique in this list.
02
Compact Context /compact
▼
After 30–45 minutes of conversation, the context becomes cluttered. Use /compact to compress the entire history into a focused summary of key decisions. Claude Code remains efficient instead of gradually losing focus.
03
Clear the Board /clear
▼
Starting a new task? Reset everything. Carrying the context of a database refactor into a frontend redesign produces confusing and contradictory code. One conversation per feature. Always.
04
Initialize Project /init
▼
Run it at the start of every new project. Claude Code scans the codebase and generates a CLAUDE.md file — a persistent context document it automatically reads in every future session. Includes project structure, tech stack, code patterns, and key architectural decisions.
05
Check Costs /cost
▼
Shows token usage for the current session. Check hourly in long sessions. AI-assisted development costs money, and surprises are never pleasant. Set a mental budget per session and verify periodically.
06
Persistent Instructions /memory
▼
Add instructions that Claude Code remembers across all sessions. "Always use TypeScript strict mode." "Always add JSDoc comments to public functions." "Always run tests after modifying files in /src/core." These rules apply automatically without having to repeat them.
07
Terminal Integration ! prefix
▼
Prefix any message with ! to execute it as a terminal command instead of sending it to Claude. A quick way to run tests, check git status, or navigate directories without leaving the interface.
08
Multi-Model Switching Opus / Sonnet
▼
Use Opus for planning and architectural decisions. Switch to Sonnet for implementation and execution. Opus thinks deeper but costs more. Sonnet executes faster at a lower cost. Plan with the thinker. Build with the builder.

Productivity Techniques
09 – 18
09
Reference File Technique
▼
Instead of describing the code style you want, point to an existing file: "Look at how authentication is implemented in src/auth/login.ts. Implement password reset following the exact same patterns." Claude Code reads the reference and replicates patterns precisely — producing much more consistent code than verbal descriptions.
10
Debug with Screenshot Ctrl+V
▼
UI looks wrong? Don't write a paragraph. Take a screenshot. Paste it with Ctrl+V. Say: "The button isn't aligned with the input field. The spacing between cards is inconsistent. Fix both." Visual feedback is faster and more accurate than written descriptions.
11
Test-First Workflow
▼
"Write tests for a function that calculates discounted prices. Cover: normal discounts, zero discount, 100% discount, negative prices, and string input. Then implement the function to pass all tests." Tests define behavior before code exists. Implementation is automatically correct because it must pass predefined tests.
12
Incremental Build
▼
Never say "build the whole feature." Break it into steps: 1. Create database schema → test. 2. Build API endpoint → test. 3. Add validation → test. 4. Build frontend form → test. Five small steps with intermediate tests produce significantly better code than one massive implementation.
13
Codebase Questions
▼
Before implementing something in an unfamiliar section: "Read src/services/ and explain how data flows from API routes to the database. What patterns are used? What should I know before modifying anything here?" Understanding before building prevents architectural errors.
14
Diff Review
▼
After Claude Code makes changes: "Show me a diff of every file you modified. Explain every change in one sentence." This catches unintended changes — Claude Code sometimes "helpfully" edits files you didn't ask it to touch.
15
Paste Full Error
▼
When something breaks, copy the error message and FULL stack trace. Not a summary. Complete output. "I got this error: [paste]. Diagnose the root cause step by step before suggesting a solution." The step-by-step constraint stops Claude from Jumping to a wrong solution.
16
Undo Checkpoint git
▼
Before any major modification: git add . && git commit -m 'checkpoint before [change]'. If Claude Code breaks something, you go back in seconds instead of spending thirty minutes debugging what was working.
17
Parallel Session
▼
For large features, open two terminal windows. One runs Claude Code for backend implementation. The other for frontend. Each session has a clean context focused on its domain. Connect the pieces at the end.
18
The Documentation Pass
▼
After completing a feature: "Read every file you created or modified for this feature. Generate complete documentation: what each function does, how they connect, expected input/output, and non-obvious design decisions." Documentation generated right after building is more accurate than written days later.

Architecture Techniques
19 – 26
19
Architectural Audit
▼
"Analyze my requirements: [list them]. Propose 2 different architectural approaches. For each: component diagram, pros, cons, estimated complexity, and what could go wrong. Recommend one with clear reasoning." Making architectural decisions with AI analysis prevents costly rewrites later.
20
Dependency Check
▼
Before adding any new package: "I want to add [package] to handle [use case]. Verify: is it actively maintained? Are there known security issues? What's the impact on bundle size? Are there lighter alternatives for my specific case?" Prevents project bloat.
21
Pattern Enforcer CLAUDE.md
▼
Add to your CLAUDE.md: "When creating new files, follow these patterns: API routes follow the structure in src/api/example-route.ts. Database queries use the repository pattern in src/repositories/." Claude Code reads this and automatically replicates your patterns in every new file.
22
Migration Builder
▼
"I need to modify the user table: add 'role' column (enum: admin, editor, viewer) and rename 'name' to 'display_name'. Generate the migration file, update the repository layer, API routes, and TypeScript types. Show me every file that must change before making any modification." Schema changes that touch multiple layers are where Claude Code excels.
23
API Design Review
▼
"Review my API design: [paste route definitions]. Check for: inconsistent naming, missing error responses, endpoints that should have pagination, missing authentication on protected routes, REST convention violations. Suggest specific improvements."
24
Security Scan
▼
"Scan this codebase for vulnerabilities: SQL injection, XSS, exposed secrets in code, missing input validation, insecure direct object references, missing rate limiting. For every problem found: severity, exact location, why it's dangerous, and the solution."
25
Performance Profiler
▼
"Analyze this codebase for performance issues: N+1 database queries, missing indexes, unnecessary re-renders in React, large bundle imports that could be lazy-loaded, API endpoints that should be cached. Prioritize by estimated impact."
26
Refactoring Planner
▼
"Read src/services/user-service.ts. This file has grown to 800 lines and handles too many responsibilities. Propose a refactoring plan to split it into focused modules. Show proposed structure, what goes where, and verify no external imports break. DON'T start refactoring — just show me the plan."

Advanced Techniques
36 – 42
36
Autopilot Mode autopilot:
▼
Write autopilot: before any request to activate fully autonomous end-to-end execution. Claude decomposes the task, delegates to the right specialized agents, verifies results, and delivers working code without manual intervention. Example: autopilot: build a REST API for user management with JWT authentication. Everything else is automatic.
37
Ralph Mode — Never Stops ralph:
▼
The keyword ralph: activates a persistent self-referential loop: Claude doesn't stop until an architect agent verifies that the task is truly complete and correct. Ideal for complex refactors or hard bugs where halfway solutions aren't enough. Like Sisyphus — the boulder never stops rolling. Example: ralph: refactor the authentication system.
38
Ultrawork — Maximum Parallelism ulw
▼
The keyword ulw (ultrawork) launches multiple agents in parallel on the same task: 3–5x faster on multi-component implementations. Claude distributes work automatically, each agent operates on an isolated domain, results are synthesized. Example: ulw fix all TypeScript errors in the project. Combined with ralph, it becomes the most powerful mode available.
39
Deep Interview — Clarify Before Building /deep-interview
▼
When you have a vague idea, use /deep-interview "vague description" before starting to build. Claude uses Socratic questions to discover hidden assumptions, unconsidered edge cases, and implicit requirements. The result is a clear spec with a measured clarity score. Building on ambiguous requirements is the main cause of rewrites — this command eliminates it.
40
Team Mode — N Coordinated Agents team N:executor
▼
The keyword team N:executor launches N coordinated agents with a shared task list and real-time messages. Replaces manual parallel sessions with an orchestrated system: each agent knows the state of others and doesn't duplicate work. Example: team 4:executor implement all missing tests. Ideal for large tasks with independent subtasks.
41
Eco Mode — Save 40% Tokens eco:
▼
The keyword eco: activates intelligent model routing: Haiku for simple lookups and repetitive tasks, Sonnet for standard work, Opus only where deep reasoning is needed. Real savings of 30–50% on token cost without perceptible quality loss. Use for long sessions, database migrations, or any task with many repetitive steps. Example: eco: migrate all React components from class to functional.
42
Rate Limit Wait — Seamless Sessions omc wait
▼
When you reach the API rate limit, instead of losing context and starting over, use omc wait --start to start a tmux daemon that monitors limit status and automatically resumes the session when it resets. omc wait without flags shows current status and estimated recovery time. omc wait --stop disables the daemon. Requires tmux installed. Essential for long development sessions with Autopilot or Ralph active.

Deep Workflows
43 – 50
43
Hooks — Deterministic Automation PostToolUse
▼
Hooks are shell commands that run automatically at precise points in the Claude Code cycle: PreToolUse (before every tool call), PostToolUse (after every edit), Stop (before Claude finishes). Configure them in .claude/settings.json. Practical examples: auto-format with Prettier every saved file, run TypeScript type-check after every edit, block dangerous commands like rm -rf before they run. Hooks are the only deterministic automation — they don't depend on AI, they always trigger.
44
Skip Permissions — Total Automation Mode --dangerously-skip-permissions
▼
Claude Code asks for confirmation for every operation: editing a file, running a command, reading a folder. In long automation sessions, this is paralyzing. Launch Claude with claude --dangerously-skip-permissions to eliminate every confirmation prompt. The name sounds dangerous but actual risk is low if you use git checkpoints beforehand and work in an isolated worktree. Ideal for CI pipelines, Docker containers, or autonomous night sessions where you can't supervise every step.
45
Git Worktrees — Parallel Branches Without Conflicts --worktree
▼
Multiple Claude Code sessions on the same repo step on each other: conflicting edits, dirty git state, broken builds. The solution is claude --worktree branch_name which automatically creates an isolated git worktree — a separate directory linked to the same .git but with its own file tree. Each session has full ownership of its space without interfering with others. You can add --tmux to open it automatically in a new tmux pane. Result: 3-4 features developed in parallel, zero conflicts.
46
Non-Interactive Mode — Claude in CI and Scripts claude -p
▼
claude -p "prompt" runs Claude in non-interactive mode: no session, no confirmations, direct response over stdout. Integrable everywhere: GitHub Actions pipelines, pre-commit hooks, bash scripts, webhooks. Add --output-format json for output parsable by other scripts, or --output-format stream-json for real-time processing. Examples: automatic analysis of error logs at every deploy, changelog generation from git diff, automatic review of every open PR. Non-interactive mode transforms Claude from interactive tool into autonomous API.
47
Handoff Document — Context Surviving Sessions HANDOFF.md
▼
Before using /compact or opening a new session, ask: "Write a HANDOFF.md file with: what we were building, what you tried, what worked, what didn't, and the exact next steps. The next agent will read only this file." Then start a fresh session passing only the file path. Claude reads it and picks up where you left off with zero quality degradation. Update the document at every subsequent session. Critical context is never lost to auto-compaction or session timeout again.
48
Fork Conversation — Experiment Without Risks /fork
▼
When you want to try a different approach from a precise point in the conversation without losing the original thread, use /fork from inside the session or claude -c --fork-session from outside. Creates a copy of the conversation with new UUIDs: you can explore an alternative direction in the copy while the original branch remains intact. Ideal when Claude takes a wrong turn and you want to restart from a specific checkpoint without losing all prior context.
49
Container Worker — Claude Controlling Claude Docker + tmux
▼
The most advanced pattern: your local Claude Code session controls another Claude Code instance running inside a Docker container. Your session uses tmux send-keys to send prompts to the container and capture-pane to read results. The internal worker runs with --dangerously-skip-permissions. Uses: execute experimental and long tasks without approving every action, orchestrate different models (Codex, Gemini CLI) from the same control point, isolate risky tasks in sandboxes. If the worker breaks something, the container is the only one to suffer.
50
Automated Git Bisect — Find Bug Commit git bisect + tmux
▼
When a bug appeared "at some point" without knowing when: ask Claude to automate git bisect using tmux as control layer. Claude creates a testing tmux session, sends commands to verify every commit, reads result, and repeats until it isolates the culprit commit. Fully automatic, zero manual interaction. Needs a test script returning exit code 0 (ok) or 1 (bug present) — Claude can write it for you before starting bisection. Technique that transforms manual debug hours into minutes.

51
/loop and /schedule — Self-Running Tasks /loop
▼
/loop schedules a recurring prompt that Claude executes autonomously for up to 3 days. Real examples: /loop check all my open PRs. Auto-fix broken builds and when comments arrive use a worktree agent to reply. Or: /loop every morning use Slack MCP to give me summary of posts where I'm tagged. /schedule works the same for one-shot future tasks. Together they transform Claude from reactive assistant into proactive agent working while you sleep.
52
/btw — Mid-Task Questions Without Interrupting Flow /btw
▼
When Claude is working on a long task and you want to ask a question without breaking its flow, use /btw your question. Claude replies to question then resumes exactly where it stopped, without losing context of the ongoing implementation. Without /btw every question risks deviating the session or making the model lose focus. With /btw you can actively supervise a long task without sabotaging it — ask clarifying questions, verify choices, approve directions — while Claude keeps building.
53
Custom Agents — Role-Specialized Agents .claude/agents/
▼
Create personalized agents by putting .md files in .claude/agents/ folder. Every agent has name, color, set of permitted tools, dedicated model and system prompt. Example read-only agent for code review that can't edit files. Example security-only agent with access only to Read and Grep. Launch agent with claude --agent=name or add "agent": "name" in settings.json to make it default. Add "use subagents" to any request to push Claude to delegate to parallel subagents — more compute on the problem.
54
Teleport — Cross-Device Sessions --teleport
▼
Working on a long task at the terminal and have to go out? claude --teleport (or /teleport from inside a session) moves the cloud session to your current device — or vice versa. You can continue the same session from phone, browser, desktop or terminal without losing anything. /remote-control allows controlling a local session from phone or web. Available on Pro, Max, Team, Enterprise plans from CLI v2.1.51 onwards. Context, history and files stay exactly where you left them.
55
Voice Input — Talk to Claude Code /voice
▼
Claude Code has native voice input: type /voice to activate. Talking is 3–5x faster than typing for most people, especially for long prompts and complex descriptions. Claude is smart enough to interpret imperfect transcriptions — even wrong words are understood from context. For maximum control: tools like SuperWhisper or MacWhisper run locally with Whisper models and integrate to any app. Using voice in office is simpler than expected — just a headset and speak quietly. Works also on planes.
56
/install-github-app — Automatic Review on Every PR /install-github-app
▼
Run /install-github-app once to connect Claude Code to your GitHub repo. From then on Claude automatically analyzes every open PR, finds bugs and security vulnerabilities, and leaves inline comments directly on the pull request — without you doing anything. Key is customizing claude-code-review.yml: by default it's verbose commenting every detail. With a focused prompt ("report only bugs and vulnerabilities, be concise") signal becomes very high. With AI generating more code PR volume increases — this automation scales with you.
57
Lazy-Load MCP — Recover 30% Context Window ENABLE_TOOL_SEARCH
▼
Every configured MCP server loads tool definitions into every conversation from start — even if never used. With 3–4 active MCPs you can lose 30–50k context window tokens before writing a line. Solution is adding "ENABLE_TOOL_SEARCH": "true" to ~/.claude/settings.json env block. Claude will load MCP tools only when truly needed, on-demand. From v2.1.7 this happens automatically if tool descriptions exceed 10% of context. Pair this with CLAUDE_CODE_SUBAGENT_MODEL: "haiku" to prioritize cost of mechanical subagents.

58
System Prompt Patching — -50% Overhead system-prompt/
▼
Claude Code's system prompt takes about 19k tokens (~10% of 200k context window) before you write a word. The ykdojo/claude-code-tips repo includes scripts that patch the minified CLI bundle, reducing it all to ~9k tokens — 50% saving on static overhead. Patches cut verbose examples and redundant text keeping all essential instructions. Result: you start every session with much more space, conversations last longer without degradation. Must re-apply patches after every upgrade, but scripts include auto instructions for containers.
59
Custom Status Line — Total Visibility context-bar.sh
▼
Claude Code shows configurable status bar at bottom of screen. With context-bar.sh script (ykdojo repo) bar shows in real-time: active model, current directory, git branch, number of uncommitted files, sync state with origin, visual progress bar of tokens used (e.g., ██░░░░░░░░ 18% of 200k tokens) and last sent message. Supports 10 color themes. Use with /context for exact token breakdown. Knowing in real-time how much context remains transforms session management from reactive to proactive — you know when to compact before quality drops.
60
Gemini CLI as Minion — 1 Million Tokens Gemini CLI + tmux
▼
Claude Code has 200k context tokens. Gemini has 1 million. Use them together: put Gemini CLI in a tmux pane and have Claude delegate bulk codebase reading tasks to it. Example: Claude uses tmux send-keys to ask Gemini to read entire codebase (500+ files) and answer architectural questions — then Claude uses those answers to write code. Claude stays the coordinator and builder; Gemini becomes the reading engine for oversized datasets. Reddit Research workflow (ykdojo Tip 11) uses this same architecture for autonomous scraping and data analysis on Reddit.
61
Resume Session — Zero Context Warmup claude --resume
▼
claude --resume reopens last session exactly where you left it — all history, read files, decisions made, all intact. Without this flag every new session starts from scratch and Claude must re-read codebase (context warmup tax). Use claude --resume [session-id] to resume specific session. claude -l lists all recent sessions with ID, model used and timestamp. Paired with HANDOFF.md it becomes most powerful technique for multi-day tasks without losing flow.
62
Rules Directory — Lazy-Loaded Per-File Context .claude/rules/
▼
Instead of putting everything in CLAUDE.md (which always loads), use .claude/rules/*.md with YAML frontmatter. Every rule file has paths: field indicating which files or directories trigger it. Example: paths: ["src/api/**"] loads those rules only when Claude touches files in src/api/. Frontend rules don't pollute context when working on backend. Without frontmatter files behave like CLAUDE.md (always loaded). Result: cleaner context window, specific rules, zero overhead when not pertinent.
63
Challenge Claude — Don't Accept First Answer active challenge
▼
Three sentences that drastically improve generated code quality, from shanraisshan community. After mediocre implementation: "knowing everything you know now, throw this away and implement the elegant solution." After fix: "grill me on these changes and don't make the PR until I pass your tests." For certainty: "prove to me it works" and have Claude diff main and your branch. Claude tends to stop at first solution that works. These sentences trigger critical re-evaluation producing cleaner, robust code. AI quality is often proportional to applied pressure.
64
/gha — GitHub Actions Investigator /gha <url>
▼
Community slash command (ykdojo dx plugin): run /gha <url> with any failed GitHub Actions URL. Claude automatically investigates: analyzes error logs, checks if problem is flaky, identifies commit introducing failure, and suggests specific fixes. Zero manual log analysis. Installable via claude /dx from plugin marketplace. Paired with gh CLI to create draft PR with proposed fix is complete workflow: failed URL → analysis → PR with fix in under a minute.
65
Context Thresholds — When to Act and How /context
▼
Claude's quality degrades non-linearly as context fills up. Thresholds from FlorianBruniaux ultimate guide: 0–50% work freely. 50–70% start monitoring, avoid reading large unnecessary files. 70–90% use /compact at next stopping point: Claude starts losing precision. 90%+ /clear mandatory: replies become erratic. Check with /context for precise breakdown or use status line for continuous monitoring. Acting preventively at 70% is better than waiting for 90% degradation.

Advanced Architecture
66 – 70
66
Spec-First with AskUserQuestion — Spec Before Build SPEC.md
▼
Most recommended workflow from official Anthropic docs for complex features: "vantage to build [minimal description]. Interview me in detail using AskUserQuestion tool. Ask about technical implementation, UX, edge cases, tradeoffs. Don't ask obvious questions — dig into hard parts I might not have considered. Continue interviewing until everything is covered, then write complete spec in SPEC.md." Then open fresh session passing only SPEC.md. Session has clean context focused on implementation and reference spec. Zero ambiguity, zero rewrites.
67
Deterministic settings.json vs CLAUDE.md settings.json
▼
CLAUDE.md uses natural language — Claude can interpret, ignore, or apply it inconsistently. settings.json is deterministic: read by runtime, not model. Rule of thumb from shanraisshan community: if you want something to happen always without exception, put it in settings.json. Examples: attribution.commit: "" reliably disables "Co-Authored-By" (instead of writing "NEVER add Co-Authored-By" in CLAUDE.md and hoping). permissions.allow and permissions.deny control tools absolutely. Use CLAUDE.md for style/context. Use settings.json for critical behavior.
68
context:fork in Skills — Isolated Context context: fork
▼
When a skill performs many intermediate operations (reading files, running commands, iterating) all that noise pollutes the main session context. Add context: fork to skill YAML frontmatter to run it in isolated subagent. Main context sees only final result — no intermediate tool calls or command outputs. Skill can do all dirty work in its own context without token cost in main session. Pair with disable-model-invocation: true for side-effect skills you trigger manually with certainty.
69
/plugin — Community Marketplace /plugin
▼
Run /plugin to browse and install community plugins without manual config. Plugins bundle skills, subagents, hooks, MCP and LSP servers into one install. Notable: dx (ykdojo) bundles handoff, gha, clone and reddit-fetch. claude-forge (sangrokjung) adds 11 specialized agents, 40 commands and 15 skill workflows. Dippy uses AST-parsing to auto-approve safe bash commands and only ask for destructive ones — eliminates permission fatigue while keeping protections. TDD Guard blocks changes violating TDD principles in real-time via hooks.
70
RIPER — Ultimate Structured Workflow Research→Plan→Execute
▼
RIPER (Research, Innovate, Plan, Execute, Review) is structured workflow imposing clear phase separation. Research: Claude reads codebase without writing. Innovate: generates architectural options, no implementation. Plan: writes detailed plan with tests for every step (plan mode). Execute: implements exactly the approved plan, one step at a time. Review: second subagent evaluates implementation. Separation stops Claude from Jumping to "obvious solution" without understanding. Most reliable for complex features in large codebases.

Thinking Control
71 – 75
71
Thinking Hierarchy — 4 Levels of Reasoning ultrathink
▼
Claude Code has four reasoning levels activated via keywords in prompt, mapped to increasing thought token budgets. think (~4k tokens): basic reasoning for tasks with multiple variables. think hard or megathink (~10k tokens): deeper analysis for refactors and complex debug. think harder or ultrathink (~32k tokens): max budget for architecture, race conditions, migrations, security reviews. Keyword can be anywhere in prompt. Works only in Claude Code CLI. Reintroduced in v2.1.68 (March 2026). Costs significantly more tokens: use for problems that merit it.
72
/effort — Reasoning Depth per Session /effort xhigh
▼
/effort without arguments opens interactive slider. /effort low is fast/cheap — for renames, docs, mechanical edits. /effort medium is balanced default. /effort high activates extended thinking — for architecture, complex debug, security review. /effort xhigh (Opus 4.7 onwards) between high/max. Difference with ultrathink: /effort is persistent official setting for entire session; ultrathink is quick trigger for single prompt. Use Alt+T to toggle extended thinking mid-session. Set CLAUDE_CODE_EFFORT_LEVEL=low for cheap CI sessions.
73
/fast — 2.5x Speed at Same Quality /fast
▼
/fast activates fast mode of Opus 4.6/4.7: 2.5x faster output at same quality. When to use: quick feedback during interactive dev where latency breaks flow. When not to use: long agentic tasks where cost accumulates (fast mode costs ~6x per token), background or subagent sessions. Important: doesn't combine with opusplan — which already mixes model internally. Use fast mode directly when latency matters more than cost, disable for autonomous work.
74
Ultrathink + Plan Mode — Most Powerful Combo ultrathink + /plan
▼
Best performing community pattern: use ultrathink in planning phase with Opus, then execute with Sonnet. Example: "ultrathink: analyze this codebase and plan migration from REST to GraphQL. Consider: schema design, resolvers, caching, breaking changes, migration path." Claude spends 32k thinking tokens on plan. Then: switch to Sonnet and implement step-by-step. Quality of plan determines quality of final code — investing here has highest ROI. Combine multiple rounds for extremely complex problems.
75
Thinking Interleaved — Reasoning Between Tool Calls adaptive thinking
▼
Claude 4.6+ support adaptive thinking with interleaved thinking: Claude can reason between tool calls in agentic workflows. Model re-evaluates results of every call before deciding next step. Skip manual budgets for adaptive thinking. Result: higher correctness from first iteration. Set MAX_THINKING_TOKENS=0 to disable thoughts in speed-critical sessions.
