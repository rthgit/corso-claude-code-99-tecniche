Comandi Essenziali
01 – 08
01
Modalità Piano Shift + Tab
▼
Prima di qualsiasi implementazione, attiva la modalità piano. Claude Code analizza il tuo codebase e crea un piano architetturale senza scrivere codice. Rivedi il piano, approvalo, poi torna all'implementazione. Questa singola abitudine previene più bug di qualsiasi altra tecnica in questa lista.
02
Compatta il Contesto /compact
▼
Dopo 30–45 minuti di conversazione, il contesto diventa ingolfato. Usa /compact per comprimere tutta la cronologia in un riassunto focalizzato sulle decisioni chiave. Claude Code rimane efficiente invece di perdere gradualmente il filo.
03
Pulisci la Lavagna /clear
▼
Inizi un nuovo task? Azzera tutto. Portare il contesto di un refactoring del database in un redesign frontend produce codice confuso e contraddittorio. Una conversazione per feature. Sempre.
04
Inizializza il Progetto /init
▼
Eseguilo all'inizio di ogni nuovo progetto. Claude Code scansiona il codebase e genera un file CLAUDE.md — un documento di contesto persistente che legge automaticamente in ogni sessione futura. Include struttura del progetto, stack tecnologico, pattern di codice e decisioni architetturali chiave.
05
Controlla i Costi /cost
▼
Mostra l'utilizzo dei token per la sessione corrente. Controlla ogni ora nelle sessioni lunghe. Lo sviluppo assistito dall'AI costa denaro e le sorprese non sono mai piacevoli. Stabilisci un budget mentale per sessione e verifica periodicamente.
06
Istruzioni Persistenti /memory
▼
Aggiungi istruzioni che Claude Code ricorda in tutte le sessioni. "Usa sempre TypeScript strict mode." "Aggiungi sempre commenti JSDoc alle funzioni pubbliche." "Esegui sempre i test dopo aver modificato file in /src/core." Queste regole si applicano automaticamente senza doverle ripetere.
07
Integrazione Terminale ! prefisso
▼
Prefissa qualsiasi messaggio con ! per eseguirlo come comando da terminale invece di inviarlo a Claude. Modo rapido per eseguire test, controllare il git status o navigare le directory senza uscire dall'interfaccia.
08
Multi-Model Switching Opus / Sonnet
▼
Usa Opus per la pianificazione e le decisioni architetturali. Passa a Sonnet per l'implementazione e l'esecuzione. Opus pensa più in profondità ma costa di più. Sonnet esegue più velocemente e a costo inferiore. Pianifica con il pensatore. Costruisci con il costruttore.

Tecniche di Produttività
09 – 18
09
Tecnica del File di Riferimento
▼
Invece di descrivere lo stile di codice che vuoi, indica un file esistente: "Guarda come è implementata l'autenticazione in src/auth/login.ts. Implementa il reset password seguendo gli stessi identici pattern." Claude Code legge il riferimento e replica i pattern con precisione — produce codice molto più coerente delle descrizioni verbali.
10
Debug con Screenshot Ctrl+V
▼
Qualcosa sembra sbagliato nell'UI? Non scrivere un paragrafo. Fai uno screenshot. Incollalo con Ctrl+V. Di': "Il bottone non è allineato al campo input. La spaziatura tra le card è inconsistente. Correggili entrambi." Il feedback visivo è più rapido e accurato delle descrizioni scritte.
11
Workflow Test-First
▼
"Scrivi i test per una funzione che calcola prezzi scontati. Coprì: sconti normali, sconto zero, sconto 100%, prezzi negativi e input stringa. Poi implementa la funzione per passare tutti i test." I test definiscono il comportamento prima che il codice esista. L'implementazione è automaticamente corretta perché deve passare i test predefiniti.
12
Build Incrementale
▼
Non dire mai "costruisci l'intera feature." Spezzala in passi: 1. Crea lo schema del database → test. 2. Costruisci l'endpoint API → test. 3. Aggiungi la validazione → test. 4. Costruisci il form frontend → test. Cinque piccoli passi con test intermedi producono codice notevolmente migliore di un'implementazione massiccia.
13
Domande sul Codebase
▼
Prima di implementare qualcosa in una parte sconosciuta: "Leggi src/services/ e spiega come i dati fluiscono dalle route API al database. Che pattern vengono usati? Cosa devo sapere prima di modificare qualcosa qui?" Capire prima di costruire previene errori architetturali.
14
Revisione delle Diff
▼
Dopo che Claude Code apporta modifiche: "Mostrami una diff di ogni file che hai modificato. Spiega ogni cambiamento in una frase." Questo intercetta modifiche non volute — Claude Code a volte cambia "utilmente" file che non gli hai chiesto di toccare.
15
Incolla l'Errore Completo
▼
Quando qualcosa si rompe, copia il messaggio di errore e lo stack trace COMPLETO. Non un riassunto. L'output completo. "Ho ricevuto questo errore: [incolla]. Diagnostica la causa principale passo dopo passo prima di suggerire una soluzione." Il vincolo passo-dopo-passo impedisce a Claude di saltare a una soluzione sbagliata.
16
Il Checkpoint di Undo git
▼
Prima di ogni modifica importante: git add . && git commit -m 'checkpoint prima di [modifica]'. Se Claude Code rompe qualcosa, torni indietro in secondi invece di passare trenta minuti a debuggare ciò che funzionava.
17
Sessione Parallela
▼
Per feature grandi, apri due finestre terminale. Una esegue Claude Code per l'implementazione backend. L'altra per il frontend. Ogni sessione ha un contesto pulito e focalizzato sul suo dominio. Colleghi i pezzi alla fine.
18
Il Passaggio Documentazione
▼
Dopo aver completato una feature: "Leggi ogni file che hai creato o modificato per questa feature. Genera documentazione completa: cosa fa ogni funzione, come si collegano, input/output attesi e decisioni di design non ovvie." La documentazione generata subito dopo la costruzione è più accurata di quella scritta giorni dopo.

19
Audit Architetturale
▼
"Analizza i miei requisiti: [elencali]. Proponi 2 diversi approcci architetturali. Per ciascuno: diagramma dei componenti, pro, contro, complessità stimata e cosa potrebbe andare storto. Raccomandane uno con motivazione chiara." Prendere decisioni architetturali con analisi AI previene costose riscritture in seguito.
20
Verifica Dipendenze
▼
Prima di aggiungere qualsiasi nuovo pacchetto: "Voglio aggiungere [pacchetto] per gestire [caso d'uso]. Verifica: è mantenuto attivamente? Ci sono problemi di sicurezza noti? Qual è l'impatto sulla dimensione del bundle? Esistono alternative più leggere per il mio caso specifico?" Previene il bloat del progetto.
21
Il Pattern Enforcer CLAUDE.md
▼
Aggiungi al tuo CLAUDE.md: "Quando crei nuovi file, segui questi pattern: le route API seguono la struttura in src/api/example-route.ts. Le query al database usano il pattern repository in src/repositories/." Claude Code lo legge e replica automaticamente i tuoi pattern in ogni nuovo file.
22
Builder di Migrazioni
▼
"Devo modificare la tabella user: aggiungi colonna 'role' (enum: admin, editor, viewer) e rinomina 'name' in 'display_name'. Genera il file di migrazione, aggiorna il repository layer, le route API e i tipi TypeScript. Mostrami ogni file che deve cambiare prima di fare qualsiasi modifica." Le modifiche allo schema che toccano più layer sono dove Claude Code eccelle.
23
Revisione del Design API
▼
"Rivedi il mio design API: [incolla le definizioni delle route]. Verifica: naming inconsistente, risposte d'errore mancanti, endpoint che dovrebbero avere paginazione, autenticazione mancante su route protette, violazioni delle convenzioni REST. Suggerisci miglioramenti specifici."
24
Scansione Sicurezza
▼
"Scansiona questo codebase per vulnerabilità: SQL injection, XSS, segreti esposti nel codice, validazione input mancante, riferimenti diretti a oggetti non sicuri, rate limiting assente. Per ogni problema trovato: severità, posizione esatta, perché è pericoloso e la soluzione."
25
Profiler di Performance
▼
"Analizza questo codebase per problemi di performance: query N+1 al database, indici mancanti, re-render non necessari in React, import di bundle grandi che potrebbero essere lazy-loaded, endpoint API che dovrebbero essere cachati. Prioritizza per impatto stimato."
26
Il Pianificatore di Refactoring
▼
"Leggi src/services/user-service.ts. Questo file è cresciuto a 800 righe e gestisce troppe responsabilità. Proponi un piano di refactoring che lo divida in moduli focalizzati. Mostra la struttura proposta, cosa va dove, e verifica che nessun import esterno si rompa. NON iniziare il refactoring — mostrami solo il piano."

▼

29
Script di Setup Ambiente
▼

30
Generatore di Note di Release
▼
"Leggi il git log dall'ultimo tag. Genera release notes organizzate per: nuove feature, bug fix, miglioramenti di performance e breaking changes. Scrivi ogni voce in linguaggio comprensibile agli utenti, non gergo da sviluppatore. Formatta come entry di markdown changelog."
31
Builder di Seed del Database
▼

▼
"Questo bug è stato segnalato da un utente: [incolla la segnalazione]. Crea una riproduzione minima: i passi esatti, il comportamento atteso, il comportamento reale. Poi scrivi un test che fallisce che cattura questo bug. Poi correggi il codice per far passare il test."

L'Investigatore delle Blame
▼
"Questa funzione ha iniziato a fallire ieri. Leggi il git log di questo file nell'ultima settimana. Identifica quale commit ha probabilmente introdotto il problema e spiega cosa è cambiato. Poi suggerisci la correzione."
34
Risolutore di Conflitti tra Dipendenze
▼
"Sto ricevendo questo conflitto di dipendenze: [incolla l'errore]. Analizza il conflitto. Identifica quali pacchetti richiedono versioni conflittuali della dipendenza condivisa. Suggerisci la risoluzione che richiede meno cambiamenti e spiega i tradeoff."
35
La Modalità Recupero
▼
Quando Claude Code produce un'implementazione rotta e vai avanti e indietro da troppo tempo: "Stop. Leggi la versione funzionante originale di questo file da git. Ora guarda cosa stavamo cercando di ottenere: [ridefinisci l'obiettivo semplicemente]. Ricomincia da capo con un approccio diverso." A volte ripartire da zero è più veloce che correggere errori accumulati. Sapere quando usarlo.

Tecniche Avanzate
36 – 42
36
Modalità Autopilot autopilot:
▼
Scrivi autopilot: davanti a qualsiasi richiesta per attivare l'esecuzione end-to-end completamente autonoma. Claude decompone il task, delega ai giusti agenti specializzati, verifica i risultati e consegna codice funzionante senza intervento manuale. Esempio: autopilot: costruisci una REST API per la gestione degli utenti con autenticazione JWT. Tutto il resto è automatico.
37
Ralph Mode — Non Si Ferma ralph:
▼
La keyword ralph: attiva un loop auto-referenziale persistente: Claude non si ferma finché un architect agent non verifica che il task è davvero completato e corretto. Ideale per refactoring complessi o bug difficili dove le soluzioni a metà strada non servono. Come Sisifo — il masso non smette mai di rotolare. Esempio: ralph: refactora il sistema di autenticazione.
38
Ultrawork — Parallelismo Massimo ulw
▼
La keyword ulw (ultrawork) lancia più agenti in parallelo sullo stesso task: 3–5x più veloce sull'implementazione multi-componente. Claude distribuisce il lavoro automaticamente, ogni agente opera su un dominio isolato, i risultati vengono sintetizzati. Esempio: ulw correggi tutti gli errori TypeScript nel progetto. Combinato con ralph diventa la modalità più potente disponibile.
39
Deep Interview — Chiarisci Prima di Costruire /deep-interview
▼
Quando hai un'idea vaga, usa /deep-interview "descrizione vaga" prima di iniziare a costruire. Claude usa domande socratiche per scoprire assunzioni nascoste, edge case non considerati e requisiti impliciti. Il risultato è una specifica chiara con un punteggio di chiarezza misurato. Costruire su requisiti ambigui è la causa principale di riscritture — questo comando la elimina.
40
Team Mode — N Agenti Coordinati team N:executor
▼
La keyword team N:executor lancia N agenti coordinati con una lista di task condivisa e messaggi in real-time. Sostituisce le sessioni parallele manuali con un sistema orchestrato: ogni agente conosce lo stato degli altri e non duplica il lavoro. Esempio: team 4:executor implementa tutti i test mancanti. Ideale per task grandi con subtask indipendenti.
41
Eco Mode — Risparmia il 40% di Token eco:
▼
La keyword eco: attiva il routing intelligente dei modelli: Haiku per lookup semplici e task ripetitivi, Sonnet per il lavoro standard, Opus solo dove serve ragionamento profondo. Risparmio reale del 30–50% sul costo dei token senza perdita di qualità percepibile. Usalo per sessioni lunghe, migrazioni di database, o qualsiasi task con molti step ripetitivi. Esempio: eco: migra tutti i componenti React da class a functional.
42
Rate Limit Wait — Sessioni Senza Interruzioni omc wait
▼
Quando raggiungi il rate limit API, invece di perdere il contesto e ricominciare, usa omc wait --start per avviare un daemon tmux che monitora lo stato del limite e riprende automaticamente la sessione quando si resetta. omc wait senza flag mostra lo stato attuale e il tempo stimato di recupero. omc wait --stop disattiva il daemon. Richiede tmux installato. Essenziale per sessioni di sviluppo lunghe con Autopilot o Ralph attivi.

Workflow Profondi
43 – 50
43
Hooks — Automazione Deterministica PostToolUse
▼
Gli hooks sono comandi shell che si eseguono automaticamente in punti precisi del ciclo di Claude Code: PreToolUse (prima di ogni tool call), PostToolUse (dopo ogni modifica), Stop (prima che Claude finisca). Configurali in .claude/settings.json. Esempi pratici: auto-formattare con Prettier ogni file salvato, eseguire TypeScript type-check dopo ogni edit, bloccare comandi pericolosi come rm -rf prima che vengano eseguiti. Gli hooks sono l'unica automazione deterministica — non dipendono dall'AI, scattano sempre.
44
Skip Permissions — Modalità Automazione Totale --dangerously-skip-permissions
▼
Claude Code chiede conferma per ogni operazione: modificare un file, eseguire un comando, leggere una cartella. Nelle sessioni di automazione lunga questo è paralizzante. Lancia Claude con claude --dangerously-skip-permissions per eliminare ogni prompt di conferma. Il nome suona pericoloso ma il rischio reale è basso se usi git checkpoint prima e lavori in un worktree isolato. Ideale per pipeline CI, container Docker, o sessioni notturne autonome dove non puoi supervisionare ogni singolo passo.
45
Git Worktrees — Branch Paralleli Senza Conflitti --worktree
▼
Più sessioni di Claude Code sullo stesso repo si calpestano a vicenda: edits conflittuali, stato git sporco, build rotte. La soluzione è claude --worktree nome_branch che crea automaticamente un git worktree isolato — una directory separata collegata allo stesso .git ma con il suo albero di file. Ogni sessione ha piena ownership del proprio spazio senza interferire con le altre. Puoi aggiungere --tmux per aprirlo automaticamente in un nuovo pannello tmux. Risultato: 3-4 feature sviluppate in parallelo, zero conflitti.
46
Modalità Non-Interattiva — Claude in CI e Script claude -p
▼
claude -p "prompt" esegue Claude in modalità non-interattiva: niente sessione, niente conferme, risposta diretta su stdout. Integrabile ovunque: pipeline GitHub Actions, pre-commit hook, script bash, webhook. Aggiungi --output-format json per output parsabile da altri script, o --output-format stream-json per processing in real-time. Esempi: analisi automatica di log di errore ad ogni deploy, generazione changelog da git diff, review automatica di ogni PR aperta. La modalità non-interattiva trasforma Claude da strumento interattivo in API autonoma.
47
Handoff Document — Contesto che Sopravvive alle Sessioni HANDOFF.md
▼
Prima di usare /compact o aprire una nuova sessione, chiedi: "Scrivi un file HANDOFF.md con: cosa stavamo costruendo, cosa hai provato, cosa ha funzionato, cosa non ha funzionato, e i prossimi passi esatti. Il prossimo agente leggerà solo questo file." Poi inizia una sessione fresca passando solo il percorso del file. Claude lo legge e riprende da dove avevi lasciato con zero degrado di qualità. Aggiorna il documento a ogni sessione successiva. Il contesto critico non si perde mai più in auto-compaction o session timeout.
48
Fork della Conversazione — Sperimenta Senza Rischi /fork
▼
Quando vuoi provare un approccio diverso da un punto preciso della conversazione senza perdere il thread originale, usa /fork dall'interno della sessione, oppure claude -c --fork-session dall'esterno. Crea una copia della conversazione con nuovi UUID: puoi esplorare una direzione alternativa nella copia mentre il branch originale rimane intatto. Ideale quando Claude ha preso una strada sbagliata e vuoi ricominciare da un checkpoint specifico senza perdere tutto il contesto precedente.
49
Container Worker — Claude che Controlla Claude Docker + tmux
▼
Il pattern più avanzato: la tua sessione Claude Code locale controlla un'altra istanza di Claude Code che gira dentro un container Docker. La tua sessione usa tmux send-keys per inviare prompt al container e capture-pane per leggere i risultati. Il worker interno gira con --dangerously-skip-permissions. Usi: eseguire task sperimentali e lunghi senza approvare ogni azione, orchestrare modelli diversi (Codex, Gemini CLI) dallo stesso punto di controllo, isolare task rischiosi in sandbox. Se il worker rompe qualcosa, il container è l'unico a soffrire.
50
Git Bisect Automatizzato — Trovare il Commit del Bug git bisect + tmux
▼
Quando un bug è comparso "a un certo punto" senza sapere quando: chiedi a Claude di automatizzare git bisect usando tmux come layer di controllo. Claude crea una sessione tmux di test, ci invia comandi per verificare ogni commit, legge il risultato, e ripete finché non isola il commit colpevole. Tutto automatico, zero interazione manuale. Per funzionare hai bisogno di uno script di test che ritorni exit code 0 (ok) o 1 (bug presente) — Claude può scriverlo per te prima di iniziare la bisezione. Tecnica che trasforma ore di debug manuale in minuti.

Potenza Nascosta
51 – 57
51
/loop e /schedule — Task che Girano da Soli /loop
▼
/loop schedula un prompt ricorrente che Claude esegue in autonomia per fino a 3 giorni. Esempi reali: /loop controlla tutte le mie PR aperte. Auto-fixa i build rotti e quando arrivano commenti usa un worktree agent per rispondere. Oppure: /loop ogni mattina usa lo Slack MCP per darmi un riassunto dei post dove sono stato taggato. /schedule funziona allo stesso modo ma per task one-shot nel futuro. Insieme trasformano Claude da assistente reattivo in agente proattivo che lavora mentre dormi.
52
/btw — Domande Mid-Task Senza Interrompere il Flusso /btw
▼
Quando Claude sta lavorando su un task lungo e vuoi fare una domanda senza spezzare il suo flusso, usa /btw la tua domanda. Claude risponde alla domanda e poi riprende esattamente da dove si era fermato, senza perdere il contesto dell'implementazione in corso. Senza /btw ogni domanda rischia di deviare la sessione o di far perdere il filo al modello. Con /btw puoi supervisionare attivamente un task lungo senza sabotarlo — chiedere chiarimenti, verificare scelte, approvare direzioni — mentre Claude continua a costruire.
53
Custom Agents — Agenti Specializzati per Ruolo .claude/agents/
▼
Crea agenti personalizzati mettendo file .md nella cartella .claude/agents/. Ogni agente ha nome, colore, set di tool permessi, modello dedicato e prompt di sistema. Esempio di agente read-only per code review che non può modificare file. Esempio di agente security-only con accesso solo a Read e Grep. Lancia un agente con claude --agent=nome o aggiungi "agent": "nome" in settings.json per farne il default. Aggiungi "use subagents" a qualsiasi richiesta per spingere Claude a delegare a subagenti paralleli — più compute sul problema.
54
Teleport — Sessioni tra Dispositivi --teleport
▼
Stai lavorando su un task lungo al terminale e devi uscire? claude --teleport (oppure /teleport dall'interno di una sessione) sposta la sessione cloud sul tuo dispositivo attuale — o viceversa. Puoi continuare la stessa sessione dal telefono, dal browser, dal desktop o dal terminale senza perdere nulla. /remote-control ti permette di controllare una sessione locale dal telefono o dal web. Disponibile su piani Pro, Max, Team, Enterprise dalla CLI v2.1.51 in poi. Il contesto, la cronologia e i file rimangono esattamente dove li avevi lasciati.
55
Voice Input — Parla con Claude Code /voice
▼
Claude Code ha un input vocale nativo: digita /voice per attivarlo. Parlare è 3–5x più veloce che digitare per la maggior parte delle persone, soprattutto per prompt lunghi e descrizioni complesse. Claude è abbastanza intelligente da interpretare trascrizioni imperfette — anche parole sbagliate vengono capite dal contesto. Per chi vuole il massimo controllo: tools come SuperWhisper o MacWhisper girano in locale con modelli Whisper e si integrano a qualsiasi app. Usare la voce in ufficio è più semplice del previsto — basta un auricolare e parlare sottovoce. Funziona anche in aereo.
56
/install-github-app — Review Automatica su Ogni PR /install-github-app
▼
Esegui /install-github-app una sola volta per connettere Claude Code al tuo repository GitHub. Da quel momento Claude analizza automaticamente ogni PR aperta, trova bug e vulnerabilità di sicurezza, e lascia commenti inline direttamente sulla pull request — senza che tu debba fare nulla. La chiave è personalizzare il file claude-code-review.yml: di default è verboso e commenta ogni minuzia. Con un prompt mirato ("segnala solo bug e vulnerabilità, sii conciso") il segnale diventa altissimo. Con l'AI che genera più codice il volume di PR aumenta — questa automazione scala con te.
57
Lazy-Load MCP — Recupera il 30% del Context Window ENABLE_TOOL_SEARCH
▼
Ogni MCP server configurato carica le sue tool definitions in ogni conversazione fin dall'inizio — anche se non lo usi mai. Con 3–4 MCP attivi puoi perdere 30–50k token di context window prima di scrivere una riga. La soluzione è aggiungere a ~/.claude/settings.json la chiave "ENABLE_TOOL_SEARCH": "true" nel blocco env. Claude caricherà i tool MCP solo quando servono davvero, on-demand. Dalla versione 2.1.7 questo avviene automaticamente se le tool descriptions superano il 10% del context. Abbina questo a CLAUDE_CODE_SUBAGENT_MODEL: "haiku" per ottimizzare anche il costo dei subagenti meccanici.

Tecniche da Comunità
58 – 65
58
System Prompt Patching — -50% di Overhead system-prompt/
▼
Il system prompt di Claude Code occupa circa 19k token (~10% della context window da 200k) prima che tu scriva una parola. La repo ykdojo/claude-code-tips include script che patchano il bundle CLI minificato, riducendo il tutto a ~9k token — risparmio del 50% sull'overhead statico. Le patch tagliano esempi verbosi e testo ridondante mantenendo tutte le istruzioni essenziali. Il risultato: inizi ogni sessione con molto più spazio, le conversazioni durano più a lungo senza degrado. Bisogna riapplicare le patch ad ogni aggiornamento di Claude Code, ma gli script includono anche istruzioni per farlo in container automaticamente.
59
Status Line Personalizzata — Visibilità Totale context-bar.sh
▼

RIPER — Il Workflow Strutturato Definitivo Research→Plan→Execute
▼

Quando ritorni a una sessione dopo un'assenza lunga, Claude genera automaticamente un session recap (riassunto away): cosa è successo mentre eri via, quali task sono stati completati, dove si è fermato il lavoro. Utile per sessioni background con /loop o team mode dove agenti hanno lavorato in autonomia. Disattiva con CLAUDE_CODE_ENABLE_AWAY_SUMMARY=0 se preferisci non averlo. Abbinato a claude --resume e i session names (-n flag) costruisce un sistema completo di stato persistente: ogni sessione ha un nome, un recap al ritorno, e i subagenti aggiornano la task list condivisa. Sviluppo async su task multi-giorno senza mai perdere il filo.

Test Time Compute — Due Agenti Sullo Stesso Codice test time compute
▼
Tecnica dalla community shanraisshan che sfrutta il fatto che context window separate producono risultati migliori: un agente può introdurre bug che lo stesso modello, partendo da zero, riesce a trovare. Il pattern: Agente A implementa la feature nel suo context window. Agente B riceve solo il codice prodotto (non la conversazione) e ha il task esplicito di trovare bug, edge case, e problemi di sicurezza. Agente B non ha bias cognitivi da "l'ho scritto io". Lo stesso concetto si estende: usa un secondo Claude per fare peer review del piano prima dell'esecuzione, come se fosse uno staff engineer — "spin up a second Claude to review this plan as a staff engineer". Oppure usa cross-model: Opus scrive, Sonnet recensisce, o viceversa per prospettive diverse.

PRD-Driven Development — Il Documento come Fonte di Verità PRD + MCP
▼
Il pattern più scalabile per team: mantieni un Product Requirement Document (PRD.md) come unica fonte di verità e collega Claude Code via MCP. Con Notion MCP o un server custom, Claude ha sempre accesso alla versione più aggiornata dei requisiti senza che tu debba incollarli a ogni sessione. Claude può chiedere solo la sezione "Acceptance Criteria" o "Technical Requirements" tramite @notion:PRD-feature — non carica l'intero documento. Quando i requisiti cambiano nel PRD, la prossima sessione di Claude li vede automaticamente. Usa MCP elicitation per far sì che Claude chieda chiarimenti sulle parti ambigue del PRD invece di assumere. Il risultato: zero drift tra vision e codice prodotto. Particolarmente potente con /loop per task ricorrenti driven da PRD che si aggiornano nel tempo.

Speculative Bash Classifier — Zero Latency su Comandi Sicuri BASH_CLASSIFIER
▼
Quando BASH_CLASSIFIER=1 è attivo, Claude Code fa partire in anticipo un classifier speculativo per ogni comando bash prima ancora che Claude decida di eseguirlo. Se il classifier ritorna con alta confidenza che il comando è sicuro, l'approvazione scatta istantaneamente — zero attesa per la tua conferma. Il classifier "gareggia" contro il timeout: se risponde prima, approva automaticamente. Se è incerto, cade nel path normale di approvazione interattiva. Risultato: comandi come git status, npm test, ls, cat vengono approvati prima che tu veda la richiesta. Solo i comandi con pattern rischiosi (rm, curl, sudo, comandi con flag distruttivi) arrivano al tuo schermo. Riduce la permission fatigue senza disabilitare alcuna protezione.

env Vars Nascoste — Controllo a Basso Livello env vars
▼
Variabili d'ambiente poco documentate che cambiano il comportamento di Claude Code in modo significativo. CLAUDE_CODE_NO_FLICKER=1: rendering engine sperimentale che elimina il flickering del terminale durante lo streaming. MAX_THINKING_TOKENS=0: disabilita completamente il pensiero esteso per sessioni veloci. CLAUDE_CODE_EFFORT_LEVEL=low: effort fisso basso per tutte le sessioni avviate da quella shell — ottimo per CI. CLAUDE_CODE_SUBAGENT_MODEL=haiku: tutti i subagenti meccanici usano Haiku invece di Sonnet, risparmiando fino al 60% sui costi di sessioni con molti subagenti. DISABLE_TELEMETRY=1: disabilita la telemetria (attiva automaticamente il session recap). CLAUDE_CODE_ENABLE_AWAY_SUMMARY=0: disabilita il recap automatico al ritorno dalla sessione.

La Legge di Ferro dell'Autonomia Co-costruita meta-tecnica
▼
Dalla ricerca di Anthropic su dati longitudinali di utilizzo: il tasso di auto-approvazione degli utenti sale dal ~20% nelle prime 50 sessioni al >40% dopo 750 sessioni. Non perché Claude diventi meno rischioso — ma perché l'utente ha imparato esattamente dove fidarsi e dove no. Il sistema è progettato per trust trajectories, non stati fissi. La tecnica meta: tieni un diario delle decisioni — ogni volta che Claude fa qualcosa di sorprendentemente buono o sorprendentemente sbagliato, annotalo in CLAUDE.md nella sezione "Gotchas" (pattern dalla community shanraisshan). Ogni skill dovrebbe avere una sezione Gotchas che accumula i failure point di Claude su quel dominio nel tempo. La qualità non sale cambiando Claude — sale modellando con precisione il contesto che gli dai. Questa è la tecnica finale: investire nel contesto è il ROI più alto di tutta la lista.
